## 1. 🎯 本节内容

- 向上取整的不同写法

## 2. 🫧 评价

在 JS 中，如果我们要对运算 `n / t` 的结果进行向上取整，首先可能会想到使用 `Math.ceil(n / t)`。但是，这种写法有一定的局限性，并不具备跨语言的通用性，更为通用的写法是 `Math.floor((n + t - 1) / t)`。

## 3. 🤔 为什么在离散世界中我们需要“向上取整”？

在现实世界的许多场景中，我们无法处理“部分”单位，必须使用完整的整数单位。这就是向上取整存在的根本原因。以下是两个经典场景：

分页计算：

- 假设数据库中有 100 条数据，每页显示 25 条，那么需要 4 页。
- 但如果数据是 101 条，每页 25 条，则需要 5 页。
- 因为第 5 页虽然只有 1 条数据，但仍然需要占用一个完整的页面。

装箱问题：

- 有 10 个物品，每个箱子最多装 3 个，那么需要 4 个箱子。
- 前 3 个箱子各装 3 个，第 4 个箱子装 1 个。
- 即使最后一个箱子没装满，也需要一个完整的箱子。

## 4. 🤔 “向上取整”的数学表达是？

这些场景都可以抽象为同一个数学问题：给定总数 n 和每份容量 t，求至少需要多少份。用数学符号表示就是：

$$
\lceil \frac{n}{t} \rceil
$$

其中向上取整符号表示无论小数部分多小，都要进到下一个整数。

【补充】如果在 markdown 中书写的话，对应的 LaTex 的语法如下：

```latex
$$
\lceil \frac{n}{t} \rceil
$$
```

## 5. 🤔 既然有 `Math.ceil`，为什么算法界还执着于 `(n + t - 1) / t`？

在 JavaScript 中，`Math.ceil(n / t)` 看起来更直观，但在算法竞赛和跨语言开发中，`Math.floor((n + t - 1) / t)` 才是标准写法。这背后有三个核心原因。

### 5.1. 跨语言通用性

不同编程语言对除法的处理方式不同。在 C++、Java、Go 等语言中，整数除法会自动向下取整，没有内置的向上取整函数。

使用 `Math.ceil` 写法：

- 在 C++ 中需要写成 `(int)ceil((double)n / t)`
- 涉及整数到浮点数再到整数的类型转换
- 代码无法直接移植

使用 `(n + t - 1) / t` 写法：

- 在 C++、Java、JavaScript、Python 中写法完全一致
- 无需任何类型转换
- 一行代码可以在所有语言中直接运行

### 5.2. 纯整数运算优势

`Math.ceil` 依赖浮点运算单元，而 `(n + t - 1) / t` 完全在整数域内完成。

在以下场景中，纯整数运算至关重要：

- 嵌入式系统可能没有浮点运算单元
- 内核开发中禁用浮点运算
- 高性能计算中避免浮点开销
- 大整数运算中浮点数会丢失精度

### 5.3. 面试考点

在技术面试中，写出 `(n + t - 1) / t` 会被认为：

- 理解离散数学原理
- 有跨语言开发经验
- 基础扎实，不依赖库函数

而只写 `Math.ceil` 可能被认为只是调用了 API，没有深入理解背后的数学逻辑。

## 6. 🤔 公式 `(n + t - 1) / t` 是如何通过数学推导严格证明的？

我们要证明的核心等式是：

$$ \lceil \frac{n}{t} \rceil = \lfloor \frac{n + t - 1}{t} \rfloor $$

其中 n 和 t 都是正整数。

### 6.1. 带余除法基础

根据带余除法，任何整数 n 除以正整数 t 都可以唯一表示为：

$$ n = q \times t + r $$

其中：

- q 是商，等于 `Math.floor(n / t)`
- r 是余数，满足 0 ≤ r < t

### 6.2. 情况一：整除时（r = 0）

当 n 能被 t 整除时，余数 r 等于 0，此时 n = q × t。

左边计算：$$ \lceil \frac{n}{t} \rceil = \lceil \frac{q \times t}{t} \rceil = \lceil q \rceil = q $$

右边计算：$$ \lfloor \frac{n + t - 1}{t} \rfloor = \lfloor \frac{q \times t + t - 1}{t} \rfloor $$
$$ = \lfloor q + \frac{t - 1}{t} \rfloor $$

因为 t ≥ 1，所以 0 ≤ (t - 1) / t < 1，向下取整后小数部分消失：$$ = q + 0 = q $$

左右两边相等，整除情况得证。

### 6.3. 情况二：有余数时（1 ≤ r < t）

当 n 不能被 t 整除时，余数 r 至少为 1，此时 n = q × t + r。

左边计算：$$ \lceil \frac{n}{t} \rceil = \lceil q + \frac{r}{t} \rceil $$

因为 0 < r / t < 1，所以 q < q + r / t < q + 1，向上取整后：$$ = q + 1 $$

右边计算：$$ \lfloor \frac{n + t - 1}{t} \rfloor = \lfloor \frac{q \times t + r + t - 1}{t} \rfloor $$
$$ = \lfloor q + 1 + \frac{r - 1}{t} \rfloor $$

因为 1 ≤ r < t，所以 0 ≤ r - 1 < t - 1 < t，即 0 ≤ (r - 1) / t < 1：$$ = q + 1 + 0 = q + 1 $$

左右两边相等，有余数情况得证。

### 6.4. 结论

两种情况都证明等式成立，因此公式 `(n + t - 1) / t` 在数学上是严格正确的。

## 7. 🤔 为什么要加的是 `t - 1` 而不是 `1` 或其他数字？

这是理解该公式最关键的问题。偏移量的选择必须满足两个条件：整除时不误判，有余数时必进位。

### 7.1. 如果只加 1 会怎样？

假设我们使用 `(n + 1) / t`，当 t = 5 时：

- n = 5（整除）：(5 + 1) / 5 = 1.2，向下取整得 1，正确
- n = 6（余 1）：(6 + 1) / 5 = 1.4，向下取整得 1，错误！应该是 2

只加 1 无法保证所有有余数的情况都能进位。当余数较大时，加 1 不足以跨越到下一个整数倍。

### 7.2. 如果加 t 会怎样？

假设我们使用 `(n + t) / t`，当 t = 5 时：

- n = 5（整除）：(5 + 5) / 5 = 2，向下取整得 2，错误！应该是 1
- n = 6（余 1）：(6 + 5) / 5 = 2.2，向下取整得 2，正确

加 t 会导致整除情况被误判进位，多算了一个单位。

### 7.3. 为什么 `t - 1` 是完美的？

`t - 1` 是能满足两个条件的最大偏移量：

条件一：整除时不进位 当 n = q × t 时，分子变为 q × t + t - 1 = (q + 1) × t - 1 除以 t 后得 q + 1 - 1/t，因为 1/t > 0，所以结果小于 q + 1 向下取整后仍为 q，不会误判

条件二：有余数时必进位 当 n = q × t + r（r ≥ 1）时，分子变为 q × t + r + t - 1 = (q + 1) × t + (r - 1) 因为 r ≥ 1，所以 r - 1 ≥ 0，分子至少为 (q + 1) × t 除以 t 后至少为 q + 1，向下取整后为 q + 1，成功进位

### 7.4. 边界值验证

设 t = 5，验证所有余数情况：

| n   | 余数 r | n + 4 | (n + 4) / 5 | 向下取整 | 期望值 |
| --- | ------ | ----- | ----------- | -------- | ------ |
| 5   | 0      | 9     | 1.8         | 1        | 1      |
| 6   | 1      | 10    | 2.0         | 2        | 2      |
| 7   | 2      | 11    | 2.2         | 2        | 2      |
| 8   | 3      | 12    | 2.4         | 2        | 2      |
| 9   | 4      | 13    | 2.6         | 2        | 2      |
| 10  | 0      | 14    | 2.8         | 2        | 2      |

所有情况都正确，证明 `t - 1` 是最优选择。

## 8. 🤔 在 JavaScript 中，浮点数精度真的会影响 `Math.ceil` 的结果吗？

这是一个常见的担忧，但在大多数实际场景中，这种风险被高估了。我们需要分情况讨论。

### 8.1. 安全整数范围内的情况

JavaScript 的安全整数范围是 -2^53 + 1 到 2^53 - 1，约等于 ±9 × 10^15。

在这个范围内，所有整数都可以精确表示，`n / t` 的计算结果也是精确的。例如：

```javascript
Math.ceil(101 / 25) // 5，精确无误
Math.ceil(100 / 25) // 4，精确无误
```

对于算法题中常见的数据范围（如 n ≤ 10^9），`Math.ceil` 完全安全可靠。

### 8.2. 潜在的精度风险

风险出现在以下场景：

场景一：超大整数 当 n 超过安全整数范围时，整数本身就无法精确表示：

```javascript
const n = 9007199254740993 // 超过安全范围
// 此时 n 在内存中已经是不精确的
```

场景二：浮点运算后的结果 如果 n 和 t 是之前浮点运算的结果，可能带有微小误差：

```javascript
const n = 3 * 0.1 + 0.7 // 理论上等于 1，实际可能是 0.9999999999999999
const t = 0.1
Math.ceil(n / t) // 可能得到意外结果
```

场景三：极端边界值 在某些极端情况下，浮点误差可能导致整除判断错误：

```javascript
// 理论上应该整除，但浮点误差导致有小数部分
const result = 3.0000000000000004
Math.ceil(result) // 得到 4，但期望是 3
```

### 8.3. 何时必须用整数法

以下场景建议使用 `(n + t - 1) / t` 或 BigInt：

1. n 或 t 可能超过安全整数范围
2. 对精度要求极高，不能容忍任何误差
3. 需要与 C++、Java 等语言保持行为一致
4. 在性能敏感的场景中避免浮点运算

### 8.4. 实际建议

对于绝大多数 Web 开发场景：

- 数据在安全整数范围内时，`Math.ceil` 更可读，推荐使用
- 算法竞赛或跨语言代码中，使用 `(n + t - 1) / t` 更稳妥
- 涉及超大整数时，使用 BigInt 配合整数公式

## 9. 🤔 这个技巧还能推广到负数或非整数场景吗？

公式 `(n + t - 1) / t` 的推导基于一个关键假设：n 和 t 都是正整数。当这个假设不成立时，公式可能失效。

### 9.1. 负数场景的问题

当 n 为负数时，向上取整的行为与正数不同。

例如，设 t = 5：

- n = -5（整除）：期望向上取整为 -1
  - 公式计算：(-5 + 4) / 5 = -0.2，向下取整得 -1，正确
- n = -6（余数）：期望向上取整为 -1（因为 -6/5 = -1.2，向上取整是 -1）
  - 公式计算：(-6 + 4) / 5 = -0.4，向下取整得 -1，正确
- n = -10（整除）：期望向上取整为 -2
  - 公式计算：(-10 + 4) / 5 = -1.2，向下取整得 -2，正确
- n = -11（余数）：期望向上取整为 -2（因为 -11/5 = -2.2，向上取整是 -2）
  - 公式计算：(-11 + 4) / 5 = -1.4，向下取整得 -2，正确

看起来似乎可行？但问题在于不同语言对负数除法的定义不同：

- JavaScript：-11 / 5 = -2.2，Math.floor(-2.2) = -3
- C++：-11 / 5 = -2（向零取整）
- Python：-11 // 5 = -3（向下取整）

这导致公式在不同语言中对负数的处理结果不一致。

### 9.2. 负数的正确处理方式

如果需要处理负数，建议使用条件判断：

```javascript
function ceilDiv(n, t) {
  if (n >= 0) {
    return Math.floor((n + t - 1) / t)
  } else {
    return Math.floor(n / t) // 负数时行为不同
  }
}
```

或者直接使用 `Math.ceil(n / t)`，让语言处理负数语义。

### 9.3. 非整数场景

当 t 不是整数时，公式完全失效。因为 `t - 1` 的意义是基于整数除法的余数概念，非整数没有余数的定义。

例如，t = 2.5 时：

- `Math.ceil(7 / 2.5)` = `Math.ceil(2.8)` = 3
- `Math.floor((7 + 2.5 - 1) / 2.5)` = `Math.floor(8.5 / 2.5)` = `Math.floor(3.4)` = 3

虽然这个例子碰巧正确，但没有数学保证，不应依赖。

### 9.4. 其他取整方式的类似技巧

向下取整：本身就是整数除法的行为，无需特殊处理。

四舍五入：可以使用 `(n + t / 2) / t`，但同样只适用于正整数。

向零取整：在 JavaScript 中可用 `Math.trunc(n / t)`。

### 9.5. 结论

公式 `(n + t - 1) / t` 仅适用于正整数场景。遇到负数或非整数时，应回归使用语言内置的取整函数，或添加额外的条件判断。

## 10. 🤔 如何在分页、内存对齐和负载均衡中灵活应用这一公式？

这个公式在实际开发中有大量应用场景，以下是三个经典案例。

### 10.1. 应用场景一：分页计算

问题：已知总数据量 total 和每页大小 pageSize，求总页数。

传统写法：

```javascript
const totalPages = Math.ceil(total / pageSize)
```

整数公式写法：

```javascript
const totalPages = ((total + pageSize - 1) / pageSize) | 0
```

说明：末尾的 `| 0` 是按位或运算，用于将结果转换为整数（在 JavaScript 中除法结果可能是浮点数）。

完整示例：

```javascript
function calculatePages(total, pageSize) {
  if (total === 0) return 0
  return ((total + pageSize - 1) / pageSize) | 0
}

// 测试
calculatePages(100, 25) // 4
calculatePages(101, 25) // 5
calculatePages(0, 25) // 0（需要特殊处理）
```

### 10.2. 应用场景二：内存对齐

问题：分配内存时，需要将大小对齐到特定的字节边界（如 8 字节、16 字节对齐）。

传统写法：

```javascript
function align(size, alignment) {
  return Math.ceil(size / alignment) * alignment
}
```

整数公式写法：

```javascript
function align(size, alignment) {
  return (((size + alignment - 1) / alignment) | 0) * alignment
}
```

更优化的位运算写法（当 alignment 是 2 的幂时）：

```javascript
function align(size, alignment) {
  return (size + alignment - 1) & ~(alignment - 1)
}
```

说明：当 alignment 是 2 的幂（如 8、16、32）时，可以使用位运算加速，这是系统编程中的常见技巧。

### 10.3. 应用场景三：负载均衡

问题：将 n 个任务分配给 t 个线程，求负载最重的线程至少处理多少个任务。

答案：这就是 `Math.ceil(n / t)` 的直接应用。

示例：

```javascript
function maxLoad(tasks, threads) {
  return ((tasks + threads - 1) / threads) | 0
}

// 17 个任务，5 个线程
maxLoad(17, 5) // 4（最重的线程处理 4 个任务）

// 20 个任务，5 个线程
maxLoad(20, 5) // 4（均匀分配）

// 21 个任务，5 个线程
maxLoad(21, 5) // 5（最重的线程处理 5 个任务）
```

扩展问题：求负载最轻的线程处理多少个任务？

```javascript
function minLoad(tasks, threads) {
  return Math.floor(tasks / threads)
}
```

### 10.4. 综合案例：文件分块上传

场景：将一个大文件分成多个块上传，每个块最大大小为 chunkSize，求需要分成多少块。

```javascript
function calculateChunks(fileSize, chunkSize) {
  if (fileSize === 0) return 0
  return ((fileSize + chunkSize - 1) / chunkSize) | 0
}

// 计算每个块的起止位置
function getChunkRange(chunkIndex, fileSize, chunkSize) {
  const start = chunkIndex * chunkSize
  const end = Math.min(start + chunkSize, fileSize)
  return { start, end }
}

// 使用示例
const totalChunks = calculateChunks(10240, 1024) // 10 个块
for (let i = 0; i < totalChunks; i++) {
  const { start, end } = getChunkRange(i, 10240, 1024)
  // 上传第 i 块，范围是 [start, end)
}
```

## 11. 🤔 在代码审查（Code Review）中，我们该如何选择最适合的写法？

在团队协作中，代码的可读性和一致性往往比微小的性能差异更重要。以下是代码审查时的决策指南。

### 11.1. 决策因素一：项目语言环境

纯 JavaScript / TypeScript 项目：

- 优先使用 `Math.ceil(n / t)`
- 理由：语义清晰，团队成员容易理解
- 例外：性能敏感的核心算法模块

跨语言项目（如前后端共用算法）：

- 优先使用 `(n + t - 1) / t`
- 理由：代码可以直接复制到其他语言，减少维护成本
- 建议：添加注释说明这是向上取整

嵌入式或系统编程：

- 必须使用 `(n + t - 1) / t`
- 理由：可能没有浮点运算单元，或浮点运算成本极高

### 11.2. 决策因素二：数据范围

安全整数范围内：

- 两种写法都可以
- 优先考虑可读性

可能超出安全整数范围：

- 使用 BigInt 配合整数公式
- 示例：`(BigInt(n) + BigInt(t) - 1n) / BigInt(t)`

### 11.3. 决策因素三：团队规范

建议建立团队编码规范：

```markdown
## 向上取整规范

1. 业务代码优先使用 Math.ceil，保证可读性
2. 算法代码可使用 (n + t - 1) / t，但需添加注释
3. 涉及超大整数时使用 BigInt
4. 禁止混用两种写法，同一模块保持一致
```

### 11.4. 决策因素四：代码审查检查清单

在审查涉及向上取整的代码时，检查以下要点：

1. 正确性：是否正确处理了整除和有余数两种情况
2. 边界值：是否处理了 n = 0 的情况（通常需要特殊处理）
3. 类型安全：是否考虑了整数溢出风险
4. 一致性：同一模块内是否使用统一的写法
5. 注释：如果使用整数公式，是否有注释说明意图

### 11.5. 审查示例

不推荐的写法：

```javascript
// 没有注释，其他开发者可能不理解
const pages = ((total + size - 1) / size) | 0
```

推荐的写法：

```javascript
// 向上取整：计算总页数（等价于 Math.ceil(total / size)）
const pages = ((total + size - 1) / size) | 0
```

最佳实践：

```javascript
// 抽取为工具函数，统一处理
function ceilDiv(n, t) {
  return n === 0 ? 0 : ((n + t - 1) / t) | 0
}

const pages = ceilDiv(total, size)
```

### 11.6. 最终建议

1. 可读性优先：在业务代码中，`Math.ceil` 更直观
2. 一致性重要：同一项目内保持写法统一
3. 注释补充：使用整数公式时添加注释说明
4. 工具函数：抽取为公共函数，避免重复代码
5. 测试覆盖：确保整除和有余数的边界情况都有测试

记住，最好的代码不是最聪明的代码，而是最容易被团队成员理解和维护的代码。
